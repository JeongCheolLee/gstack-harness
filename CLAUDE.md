# CLAUDE.md — 코딩 컨벤션 & 실행 가이드

## 수동 코드 작성 금지 정책

이 리포지터리의 모든 코드는 에이전트(Claude/Codex)에 의해 작성된다.
사람은 프롬프트, 검토, 의사결정을 담당한다.

## 코딩 컨벤션

| 항목 | 규칙 |
|------|------|
| 언어 | 프로젝트에 따라 결정 (미정) |
| 로그 | 한국어 메시지, 영어 변수명 |
| 테스트 | 모든 기능은 테스트 커버리지 필수 |
| 파일 크기 | 단일 파일 300줄 이하 권장 |
| 에러 처리 | 경계(boundary)에서만 검증 |
| 의존성 | 리포지터리 내에서 완전히 추론 가능한 것 선호 |

## 아키텍처 레이어 규칙

```
Types → Config → Models → Repositories → Services → Routes/UI
```

- 역방향 의존성 금지 (린터로 강제)
- 교차 관심사(Auth, Telemetry, Feature Flags)는 Providers를 통해서만 유입

## 파이프라인 실행 규칙

### 전체 파이프라인 (14단계)
기능 요청 시:
1. `/office-hours` — 문제 정의 & 수요 검증
2. `/plan-ceo-review` — 제품 관점 검토
3. `/plan-eng-review` — 엔지니어링 관점 검토
4. `/design-consultation` — 디자인 시스템 수립
5. `/autoplan` — 자동 리뷰 파이프라인
6. **BUILD** — 구현
7. `/design-review` — 디자인 QA
8. `/review` — 코드 리뷰
9. `/qa` — QA 테스트
10. `/cso` — 보안 감사
11. `/ship` — PR 생성 & 병합
12. `/document-release` — 문서 업데이트
13. `/retro` — 회고
14. `/checkpoint` — 상태 저장

### 스킵 규칙
- 디자인 시스템 존재 시: #4 스킵
- 버그 수정: #1~#4 스킵
- 빠른 모드("fast"): BUILD → `/review` → `/qa` → `/ship`

### 각 스킬 완료 시
- 관련 문서 업데이트
- `docs/QUALITY_SCORE.md` 반영

## 황금 원칙 (Golden Rules)

1. 직접 만든 헬퍼보다 공유 유틸리티 패키지 선호
2. 경계에서 데이터 형태를 파싱 — 추측한 형태로 구축 금지
3. 구조화된 로깅 사용
4. 불변조건은 린터/테스트로 강제, 문서만으로는 불충분
5. 에이전트가 읽기 쉬운 코드 > 사람의 문체 선호
